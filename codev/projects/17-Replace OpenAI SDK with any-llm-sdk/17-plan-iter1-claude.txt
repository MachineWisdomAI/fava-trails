### 2. Critical Issues

#### Issue A: `ProviderError` exists, but retry exceptions mapping is incomplete

The plan says:
```python
RETRYABLE_EXCEPTIONS = (RateLimitError, ProviderError)
```

The current code retries on `openai.APIConnectionError`, `openai.RateLimitError`, and `openai.InternalServerError`. The any-llm-sdk exception hierarchy has no `APIConnectionError` equivalent — **connection-level errors may surface as `ProviderError` or as raw `httpx`/`aiohttp` exceptions** that won't be caught. The plan needs to specify exactly which any-llm exceptions map to "transient/retryable" and confirm that connection errors are covered. If they aren't, the retry logic silently breaks — calls that previously retried on network hiccups will now crash.

**Recommendation**: The builder should test what exception type a connection timeout/refusal raises through any-llm's unified exception layer with `ANY_LLM_UNIFIED_EXCEPTIONS=1`. It may need to also catch `AnyLLMError` broadly or a combination.

#### Issue B: `trust_gate.py` exception mapping is vague

The plan says:
> Replace `import openai` with `from any_llm.exceptions import ...`
> Update `except openai.APIStatusError` and `except openai.APIConnectionError` to use any-llm equivalents

But the any-llm-sdk **does not have `APIStatusError` or `APIConnectionError`**. The trust gate currently uses:
- `except openai.APIStatusError as e:` → accesses `e.status_code` and `e.message`
- `except openai.APIConnectionError as e:` → uses `type(e).__name__`

The plan doesn't specify **which** any-llm exceptions replace these, or whether the attribute access patterns (`e.status_code`, `e.message`) are compatible. The closest mapping would likely be:
- `openai.APIStatusError` → `ProviderError` (but does it have `.status_code`?)
- `openai.APIConnectionError` → `ProviderError` or `AnyLLMError` (unclear)

This needs to be explicit so the builder doesn't guess wrong.

#### Issue C: `httpx` removal may be premature

The plan removes `httpx>=0.28.1` from dependencies. But `httpx` might be used elsewhere in the project — or it may be a transitive dependency of `any-llm-sdk` itself. The plan should verify `httpx` isn't imported anywhere else before removing it.Good — `httpx` is not directly imported anywhere in `src/`. So removing it from `dependencies` is safe (it was only there as a transitive dep of the OpenAI SDK). Issue C is resolved.

KEY_ISSUES:
- **Exception mapping in `_retry.py`**: The plan proposes `(RateLimitError, ProviderError)` but doesn't account for connection-level errors (the old `APIConnectionError`). Any-llm-sdk has no direct equivalent — the builder needs guidance on what catches network failures.
- **Exception mapping in `trust_gate.py`**: The plan says "use any-llm equivalents" for `APIStatusError` and `APIConnectionError` but doesn't name them. Additionally, the current code accesses `e.status_code` and `e.message` on `APIStatusError` — the plan must confirm these attributes exist on the replacement exception class (likely `ProviderError`), or specify how to extract the same information.
- **Test gap for deleted functionality**: Tests like `test_missing_openrouter_key`, `test_missing_openai_key`, and `test_client_caching` will break when `_get_client()` is removed. The plan says "remove tests for openai provider routing" but doesn't address these non-routing tests or specify replacements.
- **`timeout` parameter**: The current `chat()` passes `timeout` to `client.chat.completions.create()`. The plan should specify whether `timeout` works with `acompletion()` or needs different handling.
